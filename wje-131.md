## 项目主要需求

### 项目概述

【全栈Web应用】建筑施工项目进度跟踪平台。面向建筑公司项目经理和施工团队，支持多项目并行管理、任务分解、进度上报、材料消耗统计和关键节点提醒。**本项目用于验证模型跨文件协同改动能力，禁止将多个职责合并进单一文件。**

### 核心实体（≥ 3 个，每个贯穿全栈）

1. **项目（Project）** — 字段：名称、地址、合同金额、开工日期、计划竣工日期、实际竣工日期、状态（Planning/InProgress/Delayed/Completed/Archived）、项目经理ID
   - 关联：1个项目 → 多个任务阶段、多个材料领用记录
   - 业务动作：创建项目、更新进度、归档项目
   - **贯穿全栈**：DB表 → model → service → controller → 前端API → store → 组件

2. **任务阶段（TaskPhase）** — 字段：所属项目ID、阶段名称、计划开始/结束日期、实际开始/结束日期、完成百分比、负责人ID、优先级（Low/Medium/High/Critical）、状态（Pending/InProgress/Completed/Blocked）
   - 关联：1个阶段 → 多个子任务
   - 业务动作：创建阶段、更新进度、阻塞/解除阻塞
   - **贯穿全栈**

3. **子任务（SubTask）** — 字段：所属阶段ID、任务名称、描述、负责人ID、工时估算（小时）、实际工时、状态（Todo/InProgress/Review/Done）、完成日期
   - 业务动作：分配任务、提交审核、标记完成
   - **贯穿全栈**

4. **材料（Material）** — 字段：材料名称、规格型号、单位、库存数量、单价、仓库位置、最低库存阈值
   - 关联：1种材料 → 多条领用记录
   - 业务动作：入库、出库、库存预警
   - **贯穿全栈**

5. **材料领用记录（MaterialUsage）** — 字段：材料ID、领用项目ID、领用阶段ID、领用数量、领用人ID、领用日期、用途说明
   - **贯穿全栈**

### 核心页面（≥ 4 个）

1. **项目总览仪表盘** `/dashboard` — 消费：Project + TaskPhase + MaterialUsage
   - 展示所有在建项目的整体进度条、延期预警列表、本月材料消耗TOP10
   - 使用 `<StatusBadge>` 和 `<ProgressBar>` 共享组件

2. **项目详情 - 甘特图视图** `/projects/:id/gantt` — 消费：Project + TaskPhase + SubTask
   - 以甘特图形式展示阶段和子任务的时间线，支持拖拽调整日期
   - 使用 `<StatusBadge>` 标注任务状态

3. **任务看板** `/projects/:id/board` — 消费：TaskPhase + SubTask
   - 看板分为 Todo / InProgress / Review / Done 四列，支持拖拽移动
   - 使用 `<StatusBadge>` 和 `<UserAvatar>` 共享组件

4. **材料管理** `/materials` — 消费：Material + MaterialUsage
   - 材料库存列表，支持入库/出库操作，库存低于阈值时高亮预警
   - 领用记录按项目和阶段筛选

5. **人员工时统计** `/reports/timesheet` — 消费：SubTask + TaskPhase
   - 按人员汇总工时，对比计划与实际差异，生成周报/月报
   - 使用 `<ProgressBar>` 展示工时利用率

### 横切关注点（≥ 2 个）

1. **RBAC 权限控制**
   - 角色：Admin / ProjectManager / Foreman / Worker
   - 触达文件层：DB role表 + 后端RBAC中间件 + 前端路由守卫 + 前端按钮/操作显隐 + 类型定义 roles.ts

2. **操作日志**
   - 记录所有关键操作（项目创建、进度变更、材料领用、任务状态变更）
   - 触达文件层：DB audit_log表 + 后端日志中间件 + 后端各service调用日志 + 前端操作日志页面 + 类型定义

3. **全局异常处理**
   - 触达文件层：后端统一异常中间件 + 后端各controller异常抛出 + 前端全局错误Boundary + 前端API拦截器 + 错误类型定义

### 共享枚举/常量（≥ 2 组）

1. **项目状态枚举 ProjectStatus**：Planning / InProgress / Delayed / Completed / Archived
   - 后端定义：`backend/src/types/enums.ts`
   - 前端定义：`frontend/src/types/enums.ts`
   - README 中必须列出此枚举在前后端的所有出现位置

2. **任务状态枚举 TaskStatus**：Todo / InProgress / Review / Done
   - 后端定义：`backend/src/types/enums.ts`
   - 前端定义：`frontend/src/types/enums.ts`
   - README 中必须列出此枚举在前后端的所有出现位置

3. **优先级枚举 Priority**：Low / Medium / High / Critical
   - 后端定义：`backend/src/types/enums.ts`
   - 前端定义：`frontend/src/types/enums.ts`

4. **材料单位枚举 MaterialUnit**：kg / 吨 / 件 / 箱 / 米 / 立方米
   - 后端定义：`backend/src/types/enums.ts`
   - 前端定义：`frontend/src/types/enums.ts`

### 共享前端组件清单（≥ 3 个）

1. **`<StatusBadge>`** — 消费页面：仪表盘、甘特图、任务看板、材料管理
2. **`<ProgressBar>`** — 消费页面：仪表盘、甘特图、工时统计
3. **`<UserAvatar>`** — 消费页面：甘特图、任务看板、工时统计
4. **`<EmptyState>`** — 消费页面：所有列表页

### 共享 hooks/utils 清单（≥ 2 个）

1. **`useProject`** — 封装项目详情的获取、更新、缓存逻辑，被甘特图/看板/材料管理共用
2. **`usePagination`** — 封装分页查询逻辑（页码、每页条数、总数），被所有列表页共用
3. **`formatDate`** / **`formatDuration`** — 日期和工时格式化工具函数

### 后端中间件清单（≥ 2 个）

1. **`authMiddleware`** — JWT 鉴权，解析用户身份
2. **`rbacMiddleware`** — 角色权限校验，按路由配置所需角色
3. **`auditLogMiddleware`** — 记录关键操作到审计日志表
4. **`errorHandlerMiddleware`** — 统一异常捕获与响应格式化
5. **`requestLoggerMiddleware`** — 记录请求耗时和基本信息

### 技术要求

- **英文短名（用于 Docker Compose）**：`construction-tracker`
- **前端端口**：18701（devServer） / 80（Docker）
- **后端端口**：19201（本地） / 3000（Docker内部）
- **前端**：React 18 + TypeScript + Vite
- **UI 库**：Ant Design 5
- **图表**：ECharts（甘特图、进度图）
- **后端**：NestJS + TypeScript
- **数据库**：MySQL 8.0
- **ORM**：TypeORM
- **认证**：JWT（access_token + refresh_token）
- **状态管理**：Zustand
- **路由**：React Router v6
- **构建工具**：Vite

### 部署要求

项目必须支持 Docker Compose 一键部署，且在 **任意目录名（包括中文目录名）** 下都能正常启动。

需要提供以下文件：
- 根目录 `docker-compose.yml`：编排前端、后端、数据库及所有依赖服务
- 根目录 `.env`：包含 `COMPOSE_PROJECT_NAME=construction-tracker`
- `frontend/Dockerfile`：前端项目构建与运行镜像（多阶段构建，先构建再用 Nginx 托管）
- `backend/Dockerfile`：后端项目构建与运行镜像
- `frontend/nginx.conf`：Nginx 配置，处理前端路由和 API 反向代理
- `.env.example`：所有环境变量的示例文件，附注释说明

docker-compose.yml 要求：
- **不要写 `version:` 字段**（Compose V2 已废弃）
- 顶层声明 `name: construction-tracker` 作为兜底
- 所有服务使用 `container_name` 命名容器，带 `${COMPOSE_PROJECT_NAME:-construction-tracker}` 前缀
- 数据库使用命名卷持久化
- 前端通过 Nginx 反向代理后端 API（`/api` 转发到 `http://backend:3000/`）
- 数据库配置 `healthcheck`，后端 `depends_on: { db: { condition: service_healthy } }`
- 前端端口 `${FRONTEND_PORT:-80}:80`，后端端口 `${BACKEND_PORT:-3000}:3000`
- `.env` / `.env.example` 必须包含：`COMPOSE_PROJECT_NAME`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_ROOT_PASSWORD`, `JWT_SECRET`, `FRONTEND_PORT`, `BACKEND_PORT`

自检清单（生成代码后必须确认）：
1. 在中文目录名下执行 `docker compose config --quiet` 不报错
2. 启动后 `docker compose ps` 显示全部容器 healthy
3. 浏览器访问 18701 能看到页面，`/api/...` 请求返回正常
4. README 中 Docker 启动命令是 `docker compose up -d`
